投递 / 清运时回收箱抓拍 4 张图片（开门前 + 关门后 × 箱内 + 箱外），由设备直传腾讯云 COS， 后端不承担图片带宽，且设备不回传 URL。照片 key 两端用同一约定格式 {sn}/{doorIndex}/{token}/<slot>.jpg，但 token 的产生方与建单时机按业务不同： 清运「开门即建单」——开门时后端按 cleanOrderId 算 key、预存 URL、连 key 一起下发； 投递「上传后建单」——开门只下发凭证，deliveryToken 与 key 由设备生成/自拼，后端在设备上报时复原 URL。 下图按「谁主动发起连接」标注每一步： ★ 需公网 IP（入站） 出站，无需公网 设备↔COS 公网直连

步骤方向谁主动发起是否需要后端有公网
① 申请 STS / 开门下发凭证（清运另带 4 个 key） 后端 → COS / 后端 → OneNet → 设备 后端 出站 不需要

② 业务事件上行（投递在此建单 + 复原 URL） OneNet → 后端 后端作 MQ 消费者拨出去 出站 免公网（MQ 订阅）；HTTP 推送为有公网时备选

结论：照片链路已不再需要任何「设备 → 后端」入站

相比旧设计（设备要调后端 /photo/sts 拿密钥、调 /photo/notify 回传 URL，两处都要后端公网入站）， 现在照片凭证搭车开门命令经 OneNet 下行，URL 由后端写库（清运开门时、投递上报建单时），设备不再对后端发起任何入站请求。

  • ① 申请 STS 是后端→COS 出站；开门 + cosToken 下发是后端→OneNet 出站（清运另带 4 个 key，投递仅凭证）。
  • ① 图片直传是设备↔COS 公网直连，不经后端（清运按下发 key、投递按设备自拼 key）。
  • ② 业务事件上行已采用 OneNet 北向 MQ 订阅（后端作消费者主动拨出去），同样免公网；投递在此刻建单并复原 4 个 URL 写库。

因此整条图片链路 + 事件上行均可在无公网 IP 的内网后端跑通（仍需出站到 OneNet / COS）。 仅当改用 OneNet「HTTP 推送」备选通道时，才需要后端公网回调地址。

说明：本图为 EcoBin 后端对接设计稿。清运为「开门即建单·定 key·存 URL」；投递为「上传后建单」—— token 与 key 由设备生成/自拼、后端上报时复原 URL（见 onenet-thing-model.md §8）。 原 /api/iot/photo/sts、/api/iot/photo/notify 端点与 photoReport 事件均已移除。OneNet 报文结构以联调实测为准。
